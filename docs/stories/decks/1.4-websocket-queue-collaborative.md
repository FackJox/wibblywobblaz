# Story 1.4: WebSocket Queue & Collaborative Features

## Status
Draft

## Story

**As a** DJ,
**I want** to see real-time song submissions and manage the collaborative queue,
**so that** I can incorporate guest requests while maintaining control of the mix.

## Acceptance Criteria

1. WebSocket connection established on DJ Deck load
2. Real-time updates show new submissions instantly
3. Queue displays: song title, artist, contributor name, platform icon
4. Drag-and-drop reordering of queue items
5. Remove button for inappropriate submissions
6. Auto-queue mode adds next song when deck is empty
7. Visual notification for new submissions
8. Connection status indicator with auto-reconnect
9. Queue persists in Redis for session recovery
10. Support for 50+ concurrent contributors

## Integration Verification

- IV1: WebSocket doesn't interfere with existing API calls
- IV2: Queue updates don't cause React re-render issues
- IV3: Graceful degradation if WebSocket fails

## Dev Notes

### WebSocket Architecture
Based on architecture document, using Socket.io for reliability:

```typescript
// Event Types
interface DJDeckEvents {
  // Client -> Server
  'join_session': { session_id: string };
  'queue_reorder': { items: QueueItem[] };
  'queue_remove': { item_id: string };
  'queue_load': { item_id: string, deck: 'A' | 'B' };
  
  // Server -> Client
  'new_submission': { item: QueueItem };
  'queue_updated': { queue: QueueItem[] };
  'user_joined': { count: number };
  'user_left': { count: number };
  'session_ended': { reason: string };
}
```

### Component Structure
```
/components/dj/
  QueueManager.tsx      # Main queue interface
  QueueCard.tsx         # Individual queue item
  QueueActions.tsx      # Management controls
  ConnectionStatus.tsx  # WebSocket status
/hooks/dj/
  use-websocket.ts      # WebSocket connection hook
  use-queue.ts          # Queue state management
```

### Redis Data Structure
```
Session Queue: LIST dj:session:{session_id}:queue
Session Meta: HASH dj:session:{session_id}:meta
Active Users: SET dj:session:{session_id}:users
TTL: 86400 seconds (24 hours)
```

### Performance Optimizations
- Throttle queue updates to max 1 per 100ms
- Virtual scrolling for queues > 50 items
- Batch WebSocket messages when possible
- Use MessagePack for smaller payload size
- Implement client-side caching

### Error Handling Strategy
1. Connection lost: Cache actions locally
2. Reconnect: Sync local and server state
3. Server error: Fallback to REST API
4. Rate limiting: Queue events client-side
5. Invalid data: Show error, skip item

### Scaling Considerations
- Horizontal scaling with Redis pub/sub
- Sticky sessions for WebSocket connections
- Load balancer with WebSocket support
- Connection limit per IP address
- Graceful shutdown handling
