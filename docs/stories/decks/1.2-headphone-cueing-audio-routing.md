# Story 1.2: Headphone Cueing & Audio Routing System

## Status
Draft

## Story

**As a** DJ,
**I want** to preview upcoming tracks in my headphones while the audience hears the main mix,
**so that** I can beatmatch and prepare smooth transitions.

## Acceptance Criteria

1. Cue/PFL buttons on each deck route audio to headphone output
2. Headphone mix knob blends between cue and master signals (0-100%)
3. WebAudio API splits audio into master and cue channels
4. Visual indicators show which deck is cued
5. For local files: true pre-fader listening via gain nodes
6. For streaming: secondary hidden player instances for preview
7. Browser prompts for audio output permissions when needed
8. Graceful fallback message if browser doesn't support audio routing
9. Headphone volume control independent of master volume

## Integration Verification

- IV1: Audio routing doesn't interfere with existing sound effects
- IV2: Cue system properly releases resources when switching tracks
- IV3: No audio glitches when toggling between cue and master

## Dev Notes

### Audio Routing Architecture
Based on the architecture document, the cue system will use WebAudio API's channel routing capabilities:

```typescript
// Simplified routing structure
interface CueSystem {
  masterDestination: AudioDestinationNode;
  cueDestination: MediaStreamAudioDestinationNode;
  mixNode: GainNode; // Controls cue/master blend
  cueGainNode: GainNode; // Cue volume
  masterGainNode: GainNode; // Master volume
}
```

### Browser Compatibility
**Primary Support:**
- Chrome/Edge: Full WebAudio routing support
- Firefox: Full support with some latency
- Safari Desktop: Limited but functional

**Limited Support:**
- Safari iOS: No true audio routing (use visual fallback)
- Mobile browsers: May have restrictions on multiple audio contexts

### UI Component Structure
```
/components/dj/
  HeadphoneCue.tsx       # Cue button component
  HeadphoneMixKnob.tsx   # Cue/Master blend control
  CueIndicator.tsx       # Visual cue status display
```

### Integration Points
- Extends existing audio-engine.ts from Story 1.1
- Uses existing Radix UI Slider component
- Maintains existing color scheme (#EC4899 for cue buttons)
- Integrates with existing transport controls

### Key Technical Challenges
1. **Safari iOS Limitations**: Cannot route audio to different outputs
   - Solution: Secondary player with visual sync only
   
2. **Streaming Source Cueing**: Cannot access raw audio from YouTube/SoundCloud
   - Solution: Duplicate player instances with synchronized playback

3. **Bluetooth Latency**: Significant delay on wireless headphones
   - Solution: Warning message recommending wired connection

### Performance Requirements
- Cue latency < 50ms for accurate beatmatching
- No audio glitches when switching cue states
- Support for simultaneous playback of 2 sources
- Memory usage increase < 50MB with cue active
